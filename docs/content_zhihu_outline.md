# 知乎长文：为什么你的 Postgres 会“无缘无故变慢/变脏”？

## 标题候选
1) 你以为是慢查询，其实是 schema 在慢性自杀：Postgres 结构风险清单
2) 一次事故复盘：缺一个约束，数据就开始腐烂（以及如何提前扫描出来）
3) Postgres schema review 不靠感觉：我做了个结构风险扫描器 SchemaSentry

## 文章结构
1) 引子：事故不是从报警开始的，而是从一次“随手建表”开始的
2) 5类最常见结构风险（配小例子）
   - 没PK/没唯一约束：重复与脏数据
   - FK缺失/无索引：删除更新变慢、锁
   - 金额/时间用错类型：隐蔽bug
   - JSONB滥用：CPU与可维护性
   - enum当业务状态：迁移痛苦
3) 为什么 code review 很难抓住这些？（上下文缺失、规则散落、时间紧）
4) 解决方案：把 schema 当代码做静态扫描
5) 我做的工具：SchemaSentry
   - 输入：pg_dump --schema-only
   - 输出：风险报告 + 修复建议
6) 结尾：征集 schema 样例（可脱敏），一起把误报压下去
